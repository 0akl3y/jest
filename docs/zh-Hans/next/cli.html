Jest CLI Options

jest 命令行运行器拥有诸多有用的选项。 运行jest --help可以查看所有可用的选项 你可以通过下面所示选项进行测试当然你也可以通过想要的方式进行测试。 每一条 Jest 的 配置选项 都可以通过命令行来指定.

大致简述如下：

在命令行运行

运行所有测试(默认)

<code>jest</code>

仅运行符合指定用模板或文件名的测试︰

jest my-test #or
jest path/to/my-test.js

运行基于 hg/git (未提交的文件) 修改的文件的测试︰

jest -o

运行类似 path/to/fileA.js 和 path/to/fileB.js的测试︰

jest --findRelatedTests path/to/fileA.js path/to/fileB.js

运行匹配指定spec名称的测试用例 (主要根据describe和test中的名称来匹配)

jest -t name-of-spec

运行监视模式︰

jest --watch #runs jest -o by default
jest --watchAll #runs all tests

监视模式也支持指定文件名或路径，来运行特定的测试集。

Using with yarn

If you run Jest via yarn test, you can pass the command line arguments directly as Jest arguments.

Instead of:

jest -u -t="ColorPicker"

现在你可以使用

yarn test -u -t="ColorPicker"

使用npm脚本

If you run Jest via npm test, you can still use the command line arguments by inserting a -- between npm test and the Jest arguments.

Instead of:

jest -u -t="ColorPicker"

现在你可以使用

npm test -- -u -t="ColorPicker"

参考

jest <regexForTestFiles>

当你带着一个参数运行 jest 命令，这个参数会被当做一个正则表达式来匹配项目中相应的文件。 这使得我们可以通过提供一个表达式来运行指定的测试集。 只有被匹配到的文件才会被选中和执行。 根据你的终端不同，你可能需要将这个参数加上引号: jest "my.*(complex)?pattern"。 在 Windows 平台上，对于路径分隔符，你需要将 `` 转义为 \ 来使用，或者直接使用 / 。

--bail

或： -b. 只要一有测试不通过，立马退出测试套件。

--cache

是否使用缓存。 默认值为 true。 使用 --no-cache 禁用缓存。 * 注意: 只有在遇到与缓存相关的问题时才应禁用缓存。 平均来说，禁用缓存会令 Jest 运行至少慢两倍。*

如果要检查缓存, 请使用 -showConfig 并查看 cacheDirectory 的值。如果要清除缓存, 请使用 -clearCache。

--changedFilesWithAncestor

运行与当前更改或上次提交中所做的更改相关的测试。与 -onlyChanged 的行为相似。

--changedSince

运行与指定分支的更改相关的测试。 如果当前分支偏离了给定的分支, 则只测试本地所做的更改。 与 --onlyChanged 的行为相似。

--ci

指定该参数时，Jest 会认为目前是在 CI 环境上运行。 当遇到新的快照时，这会改变原来的行为。 这种情况下当 Jest 遇到新的快照时，不会再像平时自动保存新的快照，而会将测试标记为失败，并要求带上 --updateSnapshot 参数来运行 Jest。

--clearCache

删除 Jest 的缓存目录, 然后不运行测试直接退出。 已配置的 cacheDirectory 目录或 Jest 的默认缓存目录将被删除。 通过执行 jest --showConfig 可以看到默认的缓存目录。 * 注意：清除缓存将降低性能。*

--collectCoverageFrom=<glob>

A glob pattern relative to matching the files that coverage info needs to be collected from.

--colors

即便stdout不是TTY模式， 也要强制高亮显示测试结果。

--config=<path>

或: -c. Jest配置文件的路径，用来指定如何查找和执行测试。 如果没有设置 rootDir ，当前目录将被当做项目的根目录。 也可以使用 JSON 编码的值，Jest 会将其当做配置来使用。

--coverage

告诉 Jest 收集覆盖率信息并输出。

--debug

打印关于 Jest 配置的调试信息。

--env=<environment>

测试使用的环境。可以指向任何文件或者 node 模块。 例如 jsdom, node 或者 path/to/my-environment.js。

--expand

别名︰ -e。使用该参数来对比完整的差异和错误，而非修复。

--findRelatedTests <spaceSeparatedListOfSourceFiles>

Find and run the tests that cover a space separated list of source files that were passed in as arguments. Useful for pre-commit hook integration to run the minimal amount of tests necessary. Can be used together with --coverage to include a test coverage for the source files, no duplicate --collectCoverageFrom arguments needed.

--forceExit

强制Jest在所有测试运行完后退出。 对于遇到由测试发起但无法充分清理资源的情况很有用。 注意︰ 此功能是临时救急方法。 如果 Jest 在测试运行完后没有退出，那就意味着还有外部资源被占用，或者是在你的代码中还有计时器未结束。 为了确保 Jest 可以正常结束，建议你在每个测试后都释放和关闭用到的外部资源。

--help

显示帮助信息，类似于本页文档。

--json

以 JSON 模式显示测试结果。 该模式会将所有测试输出和用户信息都发送到 stderr。

--outputFile=<filename>

通过 — — json 参数可以将测试结果写入到指定文件。

--lastCommit

运行受上次提交的文件更改影响的所有测试。与 --onlyChanged 的行为相似。

--listTests

以 JSON 的形式列出根据给定的参数将要运行的所有测试并退出。可以与 --findRelatedTests 一起使用，来得知哪些测试将会被运行。

--logHeapUsage

记录每个测试后的堆内存使用情况。主要用来调试内存泄漏。 一般与 --runInBand 和 --expose-gc 一起使用。

--maxWorkers=<num>

别名︰ -w。 设定测试会使用的最大 worker 数目。 默认会使用你的计算机上可用的内核的数量。 在类似 CI 等有资源限制的环境下需要进行相关调整时很有用。但多数场景都应该使用默认值。

--noStackTrace

禁止栈跟踪信息在测试结果输出中。

--notify

在测试结束后发送通知。当你不想专心致志等待测试结果时很有用。

--onlyChanged

或: -o. 尝试根据当前项目中哪些文件已经更改来识别应该运行哪些测试。 只有在 git/hg 项目中，并且都是静态依赖关系时有用。(例如： 没有动态的 require) 。

--passWithNoTests

允许在没有找到文件的情况下通过测试。

--projects <project1> ... <projectN>

从一个或多个项目中运行测试。

--runInBand

别名: -i. 仅在当前的进程中连续运行所有测试，而非通过创建的子进程的工作池来运行测试。在调试时很有用。

--setupTestFrameworkScriptFile=<file>

指定一个模块的路径，在每一个测试执行前运行它来配置或启动测试框架。注意：由这个设置脚本导入的文件在测试期间不会被 mock。

--showConfig

输出Jest配置，然后退出。

--silent

阻止所有测试通过控制台输出信息。

--testNamePattern=<regex>

别名︰ -t。 Run only tests with a name that matches the regex. For example, suppose you want to run only tests related to authorization which will have names like "GET /api/posts with auth", then you can use jest -t=auth.

Note: The regex is matched against the full name, which is a combination of the test name and all its surrounding describe blocks.

--testLocationInResults

将 location 字段添加到测试结果中。要在报告中记录测试的位置时很有用。

请注意 column 是从 0 开始，而 line 不是。

{
  "column": 4,
  "line": 5
}

--testPathPattern=<regex>

一个正则表达式字符串，在运行测试前匹配所有的测试文件路径。 在 Windows 平台上，对于路径分隔符，你需要将 `` 转义为 \ 来使用，或者直接使用 / 。

--testRunner=<path>

允许你指定自定义测试运行程序。

--updateSnapshot

或: -u. 在运行测试中使用这个参数来重新录制每个失败测试的快照。 还可以和测试套件模式或 --testNamePattern 一起使用，从而重新录制的快照。

--useStderr

转移所有输出到stderr(标准错误输出).

--verbose

层次显示测试套件中每个测试的结果。

--version

或: -v. 打印版本并退出。

--watch

监视文件更改，并重新运行与已更改的文件相关的测试。如果在文件发生更改时想要重新运行所有测试，可以改为使用 --watchAll 。

--watchAll

监视文件的更改并在任何更改时重新运行所有测试。若果只想重新运行已更改文件的测试，那么请使用 --watch 选项。

--watchman

是否使用 watchman 进行文件抓取。默认值为 true。使用 --no-watchman 来禁用。
