Manual pentru dubluri

Mock-urile manuale sunt folosite pentru a înlocui funcţionalități cu date mock. De exemplu, în loc să accesați o resursă din rețea precum un site sau o bază de date, aţi putea dori să creați un mock manual care vă permite să utilizaţi date false. Acest lucru asigură că testele vor fi rapide şi nu fragile.

Mock-urile manuale sunt definite prin crearea unui modul în subdirectorul __mocks__ / imediat adiacent modulului. De exemplu, pentru crea un mock pentru un modul numit user în directorul models, creaţi un fişier numit user.js şi puneți-l în directorul models/__mocks__. Dacă modulul dublat este un modul de node (de exemplu: fs), mock-ul ar trebui să fie plasat în directorul __mocks__ adiacent lui node_modules (dacă nu aţi configurat roots pentru a indica spre un alt director decât rădăcina proiectului). Exemplu:

.
├── config
├── __mocks__
│   └── fs.js
├── models
│   ├── __mocks__
│   │   └── user.js
│   └── user.js
├── node_modules
└── views

Atunci când un mock manual există pentru un anumit modul, sistemul de module din Jest va utiliza acel modul când apelați explicit jest.mock('moduleName'). Cu toate acestea, mock-urile manuale vor avea prioritate mai mare decât modulele node chiar și în cazul în care nu este apelat jest.mock('moduleName'). Pentru a renunţa la acest comportament trebuie să apelați explicit jest.unmock('moduleName') în testele care ar trebui să utilizeze implementarea reală a modulului.

Iată un exemplu unde avem un modul care furnizează un rezumat al tuturor fişierelor dintr-un anumit director.

// FileSummarizer.js
'use strict';

const fs = require('fs');

function summarizeFilesInDirectorySync(directory) {
  return fs.readdirSync(directory).map(fileName => ({
    directory,
    fileName,
  }));
}

exports.summarizeFilesInDirectorySync = summarizeFilesInDirectorySync;

Deoarece am dori ca testele noastre să nu acceseze de fapt discul (care este destul de lent şi fragil), putem crea un mock manual pentru modulul fs prin extinderea unui mock automat. Mock-ul nostru manual va implementa versiuni personalizate pentru fs construite pentru testele noastre:

// __mocks__/fs.js
'use strict';

const path = require('path');

const fs = jest.genMockFromModule('fs');

// This is a custom function that our tests can use during setup to specify
// what the files on the "mock" filesystem should look like when any of the
// `fs` APIs are used.
let mockFiles = Object.create(null);
function __setMockFiles(newMockFiles) {
  mockFiles = Object.create(null);
  for (const file in newMockFiles) {
    const dir = path.dirname(file);

    if (!mockFiles[dir]) {
      mockFiles[dir] = [];
    }
    mockFiles[dir].push(path.basename(file));
  }
}

// A custom version of `readdirSync` that reads from the special mocked out
// file list set via __setMockFiles
function readdirSync(directoryPath) {
  return mockFiles[directoryPath] || [];
}

fs.__setMockFiles = __setMockFiles;
fs.readdirSync = readdirSync;

module.exports = fs;

Acum vom scrie testul nostru:

// __tests__/FileSummarizer-test.js
'use strict';

jest.mock('fs');

describe('listFilesInDirectorySync', () => {
  const MOCK_FILE_INFO = {
    '/path/to/file1.js': 'console.log("file1 contents");',
    '/path/to/file2.txt': 'file2 contents',
  };

  beforeEach(() => {
    // Set up some mocked out file info before each test
    require('fs').__setMockFiles(MOCK_FILE_INFO);
  });

  test('includes all files in the directory in the summary', () => {
    const FileSummarizer = require('../FileSummarizer');
    const fileSummary = FileSummarizer.summarizeFilesInDirectorySync(
      '/path/to'
    );

    expect(fileSummary.length).toBe(2);
  });
});

Exemplu mock de mai sus folosește jest.genMockFromModule pentru a genera automat un mock, şi suprascrierea comportamentul implicit al acestuia. Aceasta este abordarea recomandată, dar este complet opţională. Dacă nu doriţi să utilizaţi deloc mock-ingul automat, puteţi exporta pur şi simplu propriile funcţii din fişierul mock. Un dezavantaj al mock-ingului complet manual este că acesta e manuale – ceea ce înseamnă trebuie să le actualizaţi tot manual oricând se modifică modulul real. Din acest motiv, este mai bine să utilizaţi sau să extindeți mock-ul automat atunci când funcționează pentru nevoile voastre.

Pentru a vă asigura că un mock manual şi implementarea sa reală rămân sincronizate, ar putea fi util să includeți modulul real folosind require.requireActual(moduleName) în mock-ul manual şi modificarea sa funcţii mock înainte de exportul său.

Codul pentru acest exemplu este disponibil la exemple/manual_mocks.

Folosirea cu importurile modulelor ES #

Dacă utilizaţi importurile modulelor ES atunci probabil veți pune declaraţiile import în partea de sus a fişierului de testare. Dar adesea aveți nevoie ca Jest să folosească un mock, înainte de utilizarea modulelor. Din acest motiv, Jest va arbora automat apelurile jest.mock în partea de sus a modulului (înainte de orice importuri). Pentru a afla mai multe despre acest lucru şi a-l vedea în acţiune, vizitați acest repo.
