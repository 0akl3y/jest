Jest CLI Options

A ferramenta de linha de comando do Jest possui inúmeras opções úteis. Você pode correr jest --bantu para ver todas as opções disponíveis. Muitas das opções apresentadas abaixo podem Também são usados juntos para executar o teste conforme entender. Todos de Jest's >Configuração As opções também podem ser determinadas através de CLI.

Aqui está um breve resumo:

Executando pela linha de comando

Executar todos os testes (padrão):

jest

Executar apenas testes especificados com um padrão ou nome de arquivo:

jest my-test #or
jest path/to/my-test.js

Executar testes para arquivos alterados baseado no hg/git (arquivos não commitados):

jest -o

Executar testes para os arquivos path/to/fileA.js e path/to/fileB.js:

jest --findRelatedTests path/to/fileA.js path/to/fileB.js

Execute um teste que corresponda ao nome desta especificação (corresponde ao nome no ilustra outeste, basicamente).

jest -t name-of-spec

Execute em modo "watch":

jest --watch #runs jest -o by default
jest --watchAll #runs all tests

O modo de exibição também é possível especificar um nome ou caminho para um arquivo a ser focado em um conjunto de testes específicos.

Usando com "npm scripts"

Se você executar Jest via teste npm, você ainda pode usar argumentos de linha de comando por insira um -- entre teste npm e o argumento de Jest. De:

jest -u -t="ColorPicker"

você pode usar:

npm test -- -u -t="ColorPicker"

A opção CLI tem precedência sobre o valor de Configuração.

jest <regexForTestFiles>

Quando você corre brincando Por argumento, o argumento é tratado casualmentea frase para combinar os arquivos em seu projeto. É possível executar o teste suite fornecendo um padrão. Somente os arquivos que o padrão corresponda serão apanhados e executados. Dependendo do seu terminal, talvez seja necessário citar isso argumento: jest "meu padrão (*) complexo". No Windows, você precisa usar / como separador de estrada ou fuga `` como \.

--bail

Abreviação: -b. Sai da suite de testes imediatamente assim que o primeiro teste falhar.

--cache

Se deve usar o cache. O padrão é verdadeiro. Desative o cache usando --sem-cache. Nota: o cache só deve ser desativado se você tiver experiência problemas relacionados ao cache. Em média, desabilitar o cache faz pelo menos Jest duas vezes mais lento.

Se você deseja verificar o cache, use --showConfig e eis que diretório de cache valor. Se você precisa limpar o cache, use -- clearCache.

--changedFilesWithAncestor

Execute um teste relacionado à mudança atual e a última alteração faça. Comporta se de forma semelhante a --mesclado apenas.

--mudou desde

disponível no Jest 22.2.0+

Execução de testes relacionados a mudanças desde que o ramo fornecido. Se o ramo atual se desviou de um determinado ramo, apenas as mudanças locais são feitas testado. Comporta-se de forma semelhante a --mesclado apenas.

--ci

Quando esta opção é dada, Jest assumirá que é executado no CI ambiente. Isso muda o comportamento quando é encontrado um novo "snapshot". Em vez do comportamento de rotina de armazenar novas fotos automaticamente, isso acontecerá falhou no exame e precisou Jest para correr --upedateSnapsho.

--clearCache

disponível no Jest 22.0.0+

Exclui o diretório do cache Jest e, em seguida, sai sem executar testes. Será apagar diretório de cache se a opção for ignorada ou o cache Jest padrão diretórios. O diretório de cache padrão pode ser encontrado chamando jest --showConfig. Nota: limpar o cache reduzirá o desempenho.

--collectCoverageFrom=<glob>

An array of glob patterns relative to matching the files that coverage info needs to be collected from.

--colors

Força os resultados dos testes a serem destacados mesmo se o stdout não for "TTY".

--config=<path>

Abreviação: -c. O caminho para o arquivo de configuração Jest determina como ele encontra e executa teste. Se não rootDir configurado na configuração, o diretório atual é assumido Seja o rootDir para este projeto. Isso também pode ser o valor codificado em JSON do JEST será usado como configuração.

--coverage

Indica que a informação de cobertura de teste deve ser coletada e relatada dentro de saída.

--debug

Imprima informações de depuração sobre a configuração do Jest.

--env=<environment>

O ambiente de teste é usado para todos os testes. Pode apontar para qualquer arquivo ou nó módulo. Exemplo: jsdom, nó ou caminho / para / my-environment.js.

--expand

Abreviação: -e. Use este parâmetro para mostrar erros completos ao invés de detalhes.

--findRelatedTests <spaceSeparatedListOfSourceFiles>

Encontre e execute um teste que inclua uma lista separada de arquivos de origem existentes passou como um argumento. Útil para a integração do gancho pré-commit para executar Número mínimo de testes necessários.

--forceExit

Força o Jest a fechar depois que todos os testes estiverem concluídos. É útil quando os recursos definidos pelo código de teste não podem ser limpos adequadamente. Nota: isto O recurso é escape-hatch. Se Jest não sair no final do teste, então significa que os recursos externos ainda são mantidos ou o cronômetro ainda está pendente no seu código. Recomenda-se a eliminação de recursos externos após a realização de cada teste Certifique-se de que Jest pode ser desligado de forma limpa.

--help

Exibe a informação de ajuda, similar a esta página.

--json

Imprima os resultados do teste em JSON. Este modo enviará todas as outras saídas de teste e mensagens de usuário para stderr.

--outputFile=<filename>

Escreve os resultados dos testes em um arquivo quando a opção --json também é especificada.

--lastCommit

Execute todos os testes afetados pelas alterações de arquivo no último commit feito. Behaves da mesma forma que --mesclado apenas.

--listTests

Liste todos os testes como JSON para executar Jest com argumentos e saia. Isso pode ser usado em conjunto com --findRelatedTests para saber o teste que será feito Jest corra.

--logHeapUsage

Registre o uso da pilha após cada teste. Útil para depurar vazamentos de memória. Compartilhados com --runInBand e --expose-gcno nó.

--maxWorkers=<num>

A versão comprimida: -w. Determinando o número máximo de trabalhadores, o grupo de trabalhadores colocará seus ovos para executar o teste. Este padrão é o número de núcleos disponíveis no seu computador máquinas. Pode ser útil ajustar isso em configurações com recursos limitados, como CI mas os padrões devem ser adequados para a maioria dos casos de uso.

--noStackTrace

Desabilita o rastreio da pilha na saída dos resultados dos testes.

--notify

Permite notificações para os resultados do teste. Bem, se você não quer você consciência para poder se concentrar em qualquer coisa exceto testar JavaScript.

--onlyChanged

Abreviação: -o. Tentativas de identificar quais testis serem executados com base em quais arquivos possuem mudou no repositório atual. Só funciona se você estiver executando testes em um git/hg repositório atual e requer gráficos de dependência estática (por exemplo. não necessidades dinâmicas).

--passWithNoTests

Permite que o conjunto de testes salte quando nenhum arquivo for encontrado.

--projects <project1> ... <projectN>

Execute testes de um ou mais projetos.

--runInBand

Alias: -iExecute todos os testes em série no processo atual, em vez de criar grupo de trabalhadores do processo infantil que executam o teste. Isso pode ser útil depuração.

--setupTestFrameworkScriptFile=<file>

O caminho para um módulo que executa algum código para configurar ou configurar o teste quadro antes de cada teste. Cuidado com os arquivos importados pelo script de instalação não seja zombado durante o teste.

--showConfig

Imprime sua configuração do Jest e fecha.

--silent

Evita que testes imprimam mensagens no console.

--testNamePattern=<regex>

Alias: -t. Execute apenas testes e teste de conjuntos com nomes que correspondem à regex. Por exemplo, você só deseja executar o teste relacionado a qual autorização terá um nome como "GET / api / posting dengan auth", então você pode usarjest -t=auth.

--testLocationInResults

Adicionando um localização campo para testar os resultados. Útil se você quiser denunciar localização do teste no repórter.

Observe que coluna está indexado 0 temporariamente linhanão é.

{
  "column": 4,
  "line": 5
}

--testPathPattern=<regex>

Conecte um padrão de regexp que combine todos os caminhos de teste antes de executar exame. No Windows, você precisa usar / como separador de estrada ou fuga `` como \.

--testRunner=<path>

Permite que você especifique um executador de testes personalizado.

--updateSnapshot

Abreviação: -u. Utilize esta bandeira para reescrever os instantâneos que falharam até agora julgamento. Pode ser usado em conjunto com um padrão de suite de teste ou com --testNamePattern para regravar os "snapshots".

--useStderr

Desvia toda a saída para o stderr.

--verbose

Exibe resultados de testes individuais com a hierarquia da suite de testes.

--version

Abreviação: -v. Imprime a versão e fecha.

--watch

Vigia arquivos em busca de alterações e roda novamente testes relacionados aos arquivos alterados. Se você deseja rodar novamente todos os testes quando um arquivo for modificado, use a opção --watchAll.

--watchAll

Vigia arquivos em busca de alterações e roda novamente todos os testes quando algo muda. Se você deseja rodar apenas testes que são dependentes dos arquivos alterados, use a opção --watch.

--watchman

Quer usar o guarda para rastrear o arquivo. Padrão para verdadeiro. Desativar usando - sem guardas.
