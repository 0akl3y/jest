Configuring Jest

La configuración de Jest puede ser definida en el archivo package.json de su proyecto, a través del archivo jest.config.js o a través de la opción --config <path/to/js|json>. Si desea usar su package.json para guardar las configuración de Jest, la clave "jest" deberá ser usada en el nivel superior para que de esta manera Jest pueda saber cómo encontrar sus preferencias:

{
  "name": "my-project",
  "jest": {
    "verbose": true
  }
}

O a través de JavaScript:

// jest.config.js
module.exports = {
  verbose: true,
};

Es importante notar que la configuración debe poder serializarse a un objeto JSON.

Cuando use la opción --config, el archivo JSON no debe contener una clave "jest":

{
  "bail": true,
  "verbose": true
}

Opciones

Estas opciones le dejan controlar el comportamiento de Jest en su archivo package.json. La filosofía de Jest es funcionar bien por defecto, pero a veces sólo necesitará más poder de configuración.

automock [boolean]

Por defecto: false

This option tells Jest that all imported modules in your tests should be mocked automatically. All modules used in your tests will have a replacement implementation, keeping the API surface.

Ejemplo:

// utils.js
export default {
  authorize: () => {
    return 'token';
  },
  isAuthorized: secret => secret === 'wizard',
};

//__tests__/automocking.test.js
import utils from '../utils';

test('if utils mocked automatically', () => {
  // Public methods of `utils` are now mock functions
  expect(utils.authorize.mock).toBeTruthy();
  expect(utils.isAuthorized.mock).toBeTruthy();

  // You can provide them with your own implementation
  // or just pass the expected return value
  utils.authorize.mockReturnValue('mocked_token');
  utils.isAuthorized.mockReturnValue(true);

  expect(utils.authorize()).toBe('mocked_token');
  expect(utils.isAuthorized('not_wizard')).toBeTruthy();
});

Nota: los módulos centrales, como fs, no son simulados (mocked) por defecto. Estos pueden ser simulados de manera explicita, como jest.mock('fs').

Nota: la simulación automática (automocking) tiene un costo de desempeño más notable en proyectos grandes. Consulte aquí para detalles y soluciones.

browser [boolean]

Por defecto: false

Respect Browserify's "browser" field in package.json when resolving modules. Algunos modulos exportan diferentes versiones basandose en si están operando en nodo o en un buscador.

bail [boolean]

Por defecto: false

Por defecto, Jest ejecuta todas las pruebas y presenta todos los errores dentro de la consola al finalizar. La opción bail config puede ser usada aquí para que Jest deje de ejecutar pruebas luego de la primera falla.

cacheDirectory [string]

Por defecto: "/tmp/<path>"

Indica el directorio donde Jest debe guardar el caché de la información de las dependencias del proyecto.

Jest intenta escanear su árbol de dependencia una vez (up-front) y lo coloca en caché, para de esta manera facilitar algo del rastrilleo del sistema de archivos que necesita pasar mientras se ejecutan las pruebas. Esta opción permite configurar donde es que Jest guarda este caché en disco.

clearMocks [boolean]

Por defecto: false

Borra automáticamente las llamadas a mocks e instancias entre cada prueba. Equivalent to calling jest.clearAllMocks() between each test. This does not remove any mock implementation that may have been provided.

collectCoverage [boolean]

Por defecto: false

Indicates whether the coverage information should be collected while executing the test. Because this retrofits all executed files with coverage collection statements, it may significantly slow down your tests.

collectCoverageFrom [array]

Por defecto: undefined

Arreglo de patrones que definen el conjunto de archivos para los cuales se debe recolectar información de cobertura. If a file matches the specified glob pattern, coverage information will be collected for it even if no tests exist for this file and it's never required in the test suite.

Ejemplo:

{
  "collectCoverageFrom": [
    "**/*.{js,jsx}",
    "!**/node_modules/**",
    "!**/vendor/**"
  ]
}

This will collect coverage information for all the files inside the project's rootDir, except the ones that match **/node_modules/** or **/vendor/**.

Note: This option requires collectCoverage to be set to true or Jest to be invoked with --coverage.

coverageDirectory [string]

Por defecto: undefined

El directorio donde Jest escribe los archivos de resultados del análisis de cobertura.

coveragePathIgnorePatterns [array]

Por defecto: ["/node_modules/"]

An array of regexp pattern strings that are matched against all file paths before executing the test. If the file path matches any of the patterns, coverage information will be skipped.

Estos patrones se comparan contra la ruta completa. Utilice el token de cuerda <rootDir> para incluir la ruta hacia el directorio raíz de su proyecto para prevenirlo de ignorar accidentalmente todos sus archivos en ambientes diferentes que puedan tener diferentes directorios raíz. Ejmplo: ["<rootDir>/build/", "<rootDir>/node_modules/"].

coverageReporters [array]

Por defecto: ["json", "lcov", "text"]

A list of reporter names that Jest uses when writing coverage reports. Any istanbul reporter can be used.

Note: Setting this option overwrites the default values. Add "text" or "text-summary" to see a coverage summary in the console output.

coverageThreshold [object]

Por defecto: undefined

This will be used to configure minimum threshold enforcement for coverage results. Thresholds can be specified as global, as a glob, and as a directory or file path. If thresholds aren't met, jest will fail. Thresholds specified as a positive number are taken to be the minimum percentage required. Thresholds specified as a negative number represent the maximum number of uncovered entities allowed.

For example, with the following configuration jest will fail if there is less than 80% branch, line, and function coverage, or if there are more than 10 uncovered statements:

{
  ...
  "jest": {
    "coverageThreshold": {
      "global": {
        "branches": 80,
        "functions": 80,
        "lines": 80,
        "statements": -10
      }
    }
  }
}

If globs or paths are specified alongside global, coverage data for matching paths will be subtracted from overall coverage and thresholds will be applied independently. Thresholds for globs are applied to all files matching the glob. If the file specified by path is not found, error is returned.

For example, with the following configuration:

{
  ...
  "jest": {
    "coverageThreshold": {
      "global": {
        "branches": 50,
        "functions": 50,
        "lines": 50,
        "statements": 50
      },
      "./src/components/": {
        "branches": 40,
        "statements": 40
      },
      "./src/reducers/**/*.js": {
        "statements": 90,
      },
      "./src/api/very-important-module.js": {
        "branches": 100,
        "functions": 100,
        "lines": 100,
        "statements": 100
      }
    }
  }
}

Jest will fail if:

  • El directorio de ./src/components tiene menos del 40% de cobertura en branch o statement.
  • Uno de los archivos coincidientes con el glob ./src/reducers/**/*.js tiene menos del 90% de cobertura de statement.
  • El archivo ./src/api/very-important-module.js tiene menos del 100% de cobertura.
  • Cada archivo restante combinado tiene menos del 50% de cobertura (global).

forceCoverageMatch [array]

Default: ['']

Test files are normally ignored from collecting code coverage. With this option, you can overwrite this behavior and include otherwise ignored files in code coverage.

For example, if you have tests in source files named with .t.js extension as following:

// sum.t.js

export function sum(a, b) {
  return a + b;
}

if (process.env.NODE_ENV === 'test') {
  test('sum', () => {
    expect(sum(1, 2)).toBe(3);
  });
}

You can collect coverage from those files with setting forceCoverageMatch.

{
  ...
  "jest": {
    "forceCoverageMatch": ["**/*.t.js"]
  }
}

globals [object]

Por defecto: false

Conjunto de variables globales que necesitan estar disponibles en todos los ambientes de pruebas.

For example, the following would create a global __DEV__ variable set to true in all test environments:

{
  ...
  "jest": {
    "globals": {
      "__DEV__": true
    }
  }
}

Note that, if you specify a global reference value (like an object or array) here, and some code mutates that value in the midst of running a test, that mutation will not be persisted across test runs for other test files. In addition the globals object must be json-serializable, so it can't be used to specify global functions. For that you should use setupFiles.

globalSetup [string]

Por defecto: undefined

This option allows the use of a custom global setup module which exports an async function that is triggered once before all test suites.

globalTeardown [string]

Por defecto: undefined

This option allows the use of a custom global teardown module which exports an async function that is triggered once after all test suites.

moduleFileExtensions [array]

Por defecto: ["json", "lcov", "text"]

An array of file extensions your modules use. If you require modules without specifying a file extension, these are the extensions Jest will look for.

If you are using TypeScript this should be ["js", "jsx", "json", "ts", "tsx"], check ts-jest's documentation.

moduleDirectories [array]

Por defecto: ["/node_modules/"]

An array of directory names to be searched recursively up from the requiring module's location. Setting this option will override the default, if you wish to still search node_modules for packages include it along with any other options: ["node_modules", "bower_components"]

moduleNameMapper [object<string, string>]

Por defecto: null

A map from regular expressions to module names that allow to stub out resources, like images or styles with a single module.

Modules that are mapped to an alias are unmocked by default, regardless of whether automocking is enabled or not.

Use <rootDir> string token to refer to rootDir value if you want to use file paths.

Additionally, you can substitute captured regex groups using numbered backreferences.

Ejemplo:

{
  "moduleNameMapper": {
    "^image![a-zA-Z0-9$_-]+$": "GlobalImageStub",
    "^[./a-zA-Z0-9$_-]+\\.png$": "<rootDir>/RelativeImageStub.js",
    "module_name_(.*)": "<rootDir>/substituted_module_$1.js"
  }
}

The order in which the mappings are defined matters. Patterns are checked one by one until one fits. The most specific rule should be listed first.

Note: If you provide module name without boundaries ^$ it may cause hard to spot errors. E.g. relay will replace all modules which contain relay as a substring in its name: relay, react-relay and graphql-relay will all be pointed to your stub.

modulePathIgnorePatterns [array]

Por defecto: []

An array of regexp pattern strings that are matched against all module paths before those paths are to be considered 'visible' to the module loader. If a given module's path matches any of the patterns, it will not be require()-able in the test environment.

Estos patrones se comparan contra la ruta completa. Utilice el token de cuerda <rootDir> para incluir la ruta hacia el directorio raíz de su proyecto para prevenirlo de ignorar accidentalmente todos sus archivos en ambientes diferentes que puedan tener diferentes directorios raíz. Ejemplo: ["<rootDir>/built/"].

modulePaths [array]

Por defecto: []

An alternative API to setting the NODE_PATH env variable, modulePaths is an array of absolute paths to additional locations to search when resolving modules. Use the <rootDir> string token to include the path to your project's root directory. Ejemplo: ["<rootDir>/app/"].

notify [boolean]

Por defecto: false

Activa notificaciones para los resultados de pruebas.

notifyMode [string]

Default: always

Specifies notification mode. Requires notify: true.

Modes

  • always: always send a notification.
  • failure: send a notification when tests fail.
  • success: send a notification when tests pass.
  • change: send a notification when the status changed.
  • success-change: send a notification when tests pass or once when it fails.
  • failure-success: send a notification when tests fails or once when it passes.

preset [string]

Por defecto: undefined

A preset that is used as a base for Jest's configuration. A preset should point to an npm module that exports a jest-preset.json module on its top level.

projects [array<string | projectconfig>]

Por defecto: undefined

When the projects configuration is provided with an array of paths or glob patterns, Jest will run tests in all of the specified projects at the same time. This is great for monorepos or when working on multiple projects at the same time.

{
  "projects": ["<rootDir>", "<rootDir>/examples/*"]
}

This example configuration will run Jest in the root directory as well as in every folder in the examples directory. You can have an unlimited amount of projects running in the same Jest instance.

The projects feature can also be used to run multiple configurations or multiple runners. For this purpose you can pass an array of configuration objects. For example, to run both tests and ESLint (via jest-runner-eslint) in the same invocation of Jest:

{
  "projects": [
    {
      "displayName": "test"
    },
    {
      "displayName": "lint",
      "runner": "jest-runner-eslint",
      "testMatch": ["<rootDir>/**/*.js"]
    }
  ]
}

Note: When using multi project runner, it's recommended to add a displayName for each project. This will show the displayName of a project next to its tests.

reporters [array<modulename | [modulename, options]>]

Por defecto: undefined

disponible en Jest 20.0.0+

Se utiliza esta opción para agregar reporters personalizados a Jest. Un reporter personalizado es una clase que implementa métodos onRunStart, onTestStart, onTestResult y onRunComplete que serán llamados cuando ocurra alguno de esos eventos.

Si se especifican los informes personalizados, los informes de Jest configurados por defecto serán ignorados. Para mantener los informes por defecto, default puede ser pasado como el nombre de un módulo.

Esto anulará a los reporters por defecto:

{
  "reporters": ["<rootDir>/my-custom-reporter.js"]
}

Esto usará reporters personalizados además de los reporters por defecto que Jest proporciona:

{
  "reporters": ["default", "<rootDir>/my-custom-reporter.js"]
}

Adicionalmente, los reporters personalizados pueden ser configurados pasando un objeto de options como segundo argumento:

{
  "reporters": [
    "default",
    ["<rootDir>/my-custom-reporter.js", {"banana": "yes", "pineapple": "no"}]
  ]
}

Los modulos de report personalizados deben definir una clase que tome a GlobalConfig y las opciones del reporter como argumento del creador:

Ejemplo de reporter:

// my-custom-reporter.js
class MyCustomReporter {
  constructor(globalConfig, options) {
    this._globalConfig = globalConfig;
    this._options = options;
  }

  onRunComplete(contexts, results) {
    console.log('Custom reporter output:');
    console.log('GlobalConfig: ', this._globalConfig);
    console.log('Options: ', this._options);
  }
}

module.exports = MyCustomReporter;

Los reporteros modificados también pueden forzar a Jest a salir con código non-0 devolviendo un Error desde getLastError() métodos

class MyCustomReporter {
  // ...
  getLastError() {
    if (this._shouldFail) {
      return new Error('my-custom-reporter.js reported an error');
    }
  }
}

Para una lista completa de métodos y tipos de argumento consulte Reporter en types/TestRunner.js

resetMocks [boolean]

Por defecto: false

Restablece automáticamente el estado de los mocks entre cada prueba. Igual a llamar jest.resetAllMocks() entre cada prueba. Esto dará lugar a cualquier parodia que tiene sus implementaciones falsas eliminadas pero no restaurará su implementación inicial.

resetModules [boolean]

Por defecto: false

Al estar habilitado, el registro del módulo para cada archivo de prueba se restablecerá antes de ejecutar cada prueba individual. Esto útil para aislar módulos para cada prueba para que el estado de los módulos locales no tenga conflictos entre pruebas. Esto puede ser hecho de manera programática utilizando jest.resetModules().

resolver [string]

Por defecto: undefined

disponible en Jest 20.0.0+

Permite ocupar un resolver diferente. Esta resolución debe ser un módulo de nodo que exporte una función esperando una cadena como primer argumento para la ruta a resolver y un objeto con la siguiente estructura como segundo argumento:

{
  "basedir": string,
  "browser": bool,
  "extensions": [string],
  "moduleDirectory": [string],
  "paths": [string],
  "rootDir": [string]
}

Esta función debería arrojar una ruta hacia el módulo que debería ser resuelto o arrojar un error si el módulo no puede ser encontrado.

restoreMocks[boolean]

Por defecto: false

Restablece automáticamente el estado de los mocks entre cada prueba. Igual que llamar jest.restoreAllMocks() entre cada prueba. Esto llevará a cualquier mock que tenga sus implementaciones falsas removidas y reestablece su implementación inicial.

rootDir [string]

Por defecto: La raíz del directorio conteniendo su config file de Jest, ó el package.json, ó el pwd si no se encuentra ningún package.json

El directorio raíz a utilizar por Jest para buscar tests y módulos. Si pone la configuración de Jest dentro de su package.json y desea que el directorio raíz sea la raíz de su repositorio, el valor para este parametro de configuración se enviará automáticamente al directorio del package.json.

A menudo, querrá configurar esto a 'src' ó 'lib', dependiendo de dónde en su repositorio se almacena el código.

Observa que utilizar '<rootDir>' como un token de seguridad en cualquier otra instalación de configuración basada en ruta hará referencia a este valor. Así que, por ejemplo, si desea que su entrada de configuración setupFiles apunte hacia el archivo env-setup.js en la raíz de su proyecto, usted podría configurar su valor a ["<rootDir>/env-setup.js"].

roots [array]

Por defecto: ["<rootDir>"]

Una lista de rutas a directorios que Jest usará para buscar archivos.

Hay momentos en los que solo pide que Jest busque en un único subdirectorio (tales como los casos en los que tiene src/ un directorio en el repositorio), pero no le permite acceder al resto del repositorio.

Nota: mientras que rootDir se usa la mayoría de las veces como un token a ser reusado en otras opciones de configuración, roots se usa por los internos de Jest para localizar archivos de prueba y archivos fuente. Esto aplica también cuando se busco por mocks manuales para modulo de node_modules (__mocks__ necesitará habitar en una de las roots).

Nota: Por defecto, roots tiene una sola entrada <rootDir> pero existen casos donde quizás quiera tener multiples raíces dentro de un proyecto, por ejemplo roots: <code>roots: ["<rootDir>/src/", "<rootDir>/tests/"].

runner [string]

disponible en Jest 21.0.0+

Por defecto: "jest-runner"

Esta opción le permite utilizar un corredor personalizado en lugar del corredor que trae Jest. Los ejemplos de corredores incluyen:

  • jest-runner-eslint
  • jest-runner-mocha
  • jest-runner-tsc
  • jest-runner-prettier

Para escribir un corredor de pruebas, exporte una clase la cual acepte globalConfig en el constructor, y tenga un método de runTests con la firma:

async runTests(
  tests: Array<Test>,
  watcher: TestWatcher,
  onStart: OnTestStart,
  onResult: OnTestSuccess,
  onFailure: OnTestFailure,
  options: TestRunnerOptions,
): Promise<void>

If you need to restrict your test-runner to only run in serial rather then being executed in parallel your class should have the property isSerial to be set as true.

setupFiles [array]

Por defecto: []

Son las rutas hacia los módulos que ejecuten algún código para configurar o preparar el ambiente de prueba antes de cada prueba. Debido a que cada prueba se ejecuta en su propio ambiente, estos scripts serán ejecutado en el ambiente de prueba inmediatamente antes de ejecutar el código de prueba mismo.

Cabe destacar que este código se ejecutará antes de setupTestFrameworkScriptFile.

setupTestFrameworkScriptFile [string]

Por defecto: undefined

Es la ruta hacia un módulo que ejecuta algún código para configurar o preparar el marco de referencia antes de cada prueba. Debido a que setupFiles se ejecuta antes de que el marco de referencia de la prueba se instale en el ambiente, este archivo script le presenta la oportunidad de ejecutar algún código inmediatamente luego de que el marco de referencia de la prueba haya sido instalado en el ambiente.

Por ejemplo, Jest manda varios plug-ins a jasmine que funcionan al hacer "monkey-patching" en el API jasmine. Si deseaba añadir incluso más plug-ins jasmine a la mezcla (o si quería algunos matchers personalizados a lo largo del proyecto, por ejemplo), podría hacerlo en este módulo.

snapshotSerializers [array]

Por defecto: []

Lista de rutas que apuntan a módulos de serialización de snapshots que Jest puede ocupar para pruebas de snapshot.

Jest has default serializers for built-in JavaScript types, HTML elements (Jest 20.0.0+), ImmutableJS (Jest 20.0.0+) and for React elements. Consulte el snapshot tutorial de prueba para más información.

Ejemplo de modulo serializador:

// my-serializer-module
module.exports = {
  print(val, serialize, indent) {
    return 'Pretty foo: ' + serialize(val.foo);
  },

  test(val) {
    return val && val.hasOwnProperty('foo');
  },
};

serialize es una función que serializa un valor usando plug-ins existentes.

Como ejemplo, para usar un modulo llamado my-serializer-module como un serializador, la configuración es la siguiente:

{
  ...
  "jest": {
    "snapshotSerializers": ["my-serializer-module"]
  }
}

Y finalmente, las pruebas se ven así:

test(() => {
  const bar = {
    foo: {
      x: 1,
      y: 2,
    },
  };

  expect(bar).toMatchSnapshot();
});

Snapshot producido:

Pretty foo: Object {
  "x": 1,
  "y": 2,
}

To make a dependency explicit instead of implicit, you can call expect.addSnapshotSerializer to add a module for an individual test file instead of adding its path to snapshotSerializers in Jest configuration.

testEnvironment [string]

Por defecto: "jsdom"

Ambiente de pruebas a utilizar. El entorno predeterminado en Jest es un entorno similar a un navegador a través de jsdom. Si esta construyendo un servicio de nodo, puedes usar la opción node para usar un entorno tipo nodo en su sitio.

Si algunas pruebas requieren otro ambiente, puede añadir un docblock @jest-environment.

disponible en Jest 20.0.0+
/**
 * @jest-environment jsdom
 */

test('use jsdom in this test file', () => {
  const element = document.createElement('div');
  expect(element).not.toBeNull();
});

Puede crear su propio módulo que será usado para configurar el ambiente de prueba. El módulo debe exportar una clase con métodos setup, teardown y runScript. Puede pasar también variables desde este módulo a sus suites de prueba al signarlos al objeto this.global – esto las hará disponibles en su suite de prueba como variables globales.

disponible en Jest 22.0.0+

Nota: TestEnvironment (ambiente de prueba) está dentro de una zona segura (sandboxed). Cada sala de prueba lanzará la configuración/eliminación en su propio TestEnvironment.

Ejemplo:

// my-custom-environment
const NodeEnvironment = require('jest-environment-node');

class CustomEnvironment extends NodeEnvironment {
  constructor(config) {
    super(config);
  }

  async setup() {
    await super.setup();
    await someSetupTasks();
    this.global.someGlobalObject = createGlobalObject();
  }

  async teardown() {
    this.global.someGlobalObject = destroyGlobalObject();
    await someTeardownTasks();
    await super.teardown();
  }

  runScript(script) {
    return super.runScript(script);
  }
}

// my-test-suite
let someGlobalObject;

beforeAll(() => {
  someGlobalObject = global.someGlobalObject;
});

testEnvironmentOptions [Object]

disponible en Jest 22.0.0+

Por defecto: false

Opciones de ambiente de prueba que serán pasadas a testEnvironment. Las opciones relevantes dependen del ambiente. For example you can override options given to jsdom such as {userAgent: "Agent/007"}.

testMatch [array]

disponible en Jest 19.0.0+

(default: [ '**/__tests__/**/*.js?(x)', '**/?(*.)+(spec|test).js?(x)' ])

Patrones glob que Jest usa para detectar archivos. Por defecto busca archivos .js y .jsx dentro de carpetas __tests__, así como cualquier archivo con el sufijo de .test o .spec (por ejemplo Component.test.js o Component.spec.js). También encontrará archivos llamados test.js o spec.js.

Ver el paquete micromatch para que pueda especificar detalles en los patrones.

See also testRegex [string], but note that you cannot specify both options.

testPathIgnorePatterns [array]

Por defecto: ["/node_modules/"]

Un arreglo de patrones de cuerda regexp que se comparan con todas las rutas antes de ejecutar la prueba. Si la ruta de prueba coincide con alguno de los patrones, será saltada.

Estos patrones se comparan contra la ruta completa. Utilice el token de cuerda <rootDir> para incluir la ruta hacia el directorio raíz de su proyecto para prevenirlo de ignorar accidentalmente todos sus archivos en ambientes diferentes que puedan tener diferentes directorios raíz. Ejmplo: ["<rootDir>/build/", "<rootDir>/node_modules/"].

testRegex [string]

Por defecto: (/__tests__/.*|(\\.|/)(test|spec))\\.jsx?$

Patrón utilizado por Jest para detectar archivos de prueba. By default it looks for .js and .jsx files inside of __tests__ folders, as well as any files with a suffix of .test or .spec (e.g. Component.test.js or Component.spec.js). También encontrará archivos llamados test.js o spec.js. See also testMatch [array], but note that you cannot specify both options.

A continuación se muestra una visualización de la expresión regex por defecto:

├── __tests__
│   └── component.spec.js # prueba
│   └── anything # prueba
├── package.json # no es prueba
├── foo.test.js # prueba
├── bar.spec.jsx # prueba
└── component.js # no es prueba

testRunner [string]

Por defecto: jasmine2

Esta opción permito uso de un ejecutador de prueba personalizado. El que viene por defecto es jasmine2. Un ejecutador de pruebas puede ser proporcionado especificando una ruta a una implementación de ejecutador de prueba.

El modulo que ejecuta pruebas debe exportar una función con la siguiente estructura:

function testRunner(
  config: Config,
  environment: Environment,
  runtime: Runtime,
  testPath: string,
): Promise<TestResult>;

Un ejemplo de dicha función puede ser encontrador en nuestro incluido paquete ejecutor de pruebas jasmine2.

testURL [string]

Por defecto: about:blank

Esta opción configura el URL por el ambiente jsdom. Se refleja en propiedades como location.href.

timers [string]

Por defecto: real

Configurar este valor a fake le permite usar timers fake para funciones como setTimeout. Los timers fake son útiles cuando un pedazo de código configura un timeout largo por el que no queremos esperar en una prueba.

transform [object<string, string>]

Por defecto: undefined

Mapa de expresiones regulares que apuntan a rutas de transformers. Un transformador es un módulo que proporciona una función sincrónica para transformar archivos fuente. Por ejemplo, si quería ser capaz de usar una nueva caracteristica de lenguaje en sus módulos o pruebas que no sea aún respaldada por nodo, quizás pueda enchufar uno de muchos compiladores que compilen una versión futura de JavaScript a un actual. Ejemplo: ver el ejemplo examples/typescript o el webpack tutorial.

Los ejemplos de dichos compiladores incluyen Babel, TypeScript y async-to-gen.

Nota: los transformadores se ejecutan una sola vez por archivo, a menos que el archivo haya cambiado. Durante el desarrollo de un transformador, puede ser útil ejecutar el Jest con --no-cache a menudo delete Jest's cache.

Nota: si usa el transformador babel-jest y desea usar un pre-procesador de código adicional, tenga en cuenta que cuando "transform" esta sobre-escrito de alguna manera babel-jest ya no es cargado automáticamente. Si desea usarlo para compilar código JavaScript tiene que estar definido explicitamente. Ver babel-jest plugin

transformIgnorePatterns [array]

Por defecto: ["/node_modules/"]

Un arreglo de patrones de cuerdas regexp que son comparados con todas las rutas de archivo fuente antes de las transformación. Si las rutas de prueba coinciden con alguno de los patrones, no será transformado.

Estos patrones se comparan contra la ruta completa. Utilice el token de cuerda <rootDir> para incluir la ruta hacia el directorio raíz de su proyecto para prevenirlo de ignorar accidentalmente todos sus archivos en ambientes diferentes que puedan tener diferentes directorios raíz.

Ejemplo: ["<rootDir>/bower_components/", "<rootDir>/node_modules/"].

A veces sucede (especialmente en proyectos React Native o TypeScript) que los módulos de la tercer parte son publicados sin la conversión adecuada (transpiling). Ya que todos los archivos dentro de node_modules no son convertidos por defecto, Jest no entenderá el código en estos módulos, resultando en errores de sintaxis. Para superar este inconveniente, usted puede usar transformIgnorePatterns para autorizar el listado de esos módulos. Encontrara un buen ejemplo de este caso de uso en React Native Guide.

unmockedModulePathPatterns [array]

Por defecto: []

Un arreglo de patrones de cuerdas regexp que son comparadas con todos los módulos antes de que el cargador de módulos arroje automaticamente una simulación para estos. Si la ruta de módulo coincide con cualquiera de los patrones en esta lista, no será simulado automáticamente por el cargador de módulos.

Esto es útil para módulos 'utility' comunmente usados, que son casi siempre utilizados como detalles de implementación casi todo el tiempo (como underscore/lo-dash, etc). Generalmente es mejor mantener esta lista tan pequeña como sea posible y siempre usar llamadas explicitas jest.mock()/jest.unmock() en pruebas individuales. La instalación explícita por prueba es mucho más fácil para otros lectores de la prueba para considerar sobre el entorno en el que se ejecutará la prueba.

Es posible anular esta configuración en pruebas individuales llamando explicitamente a jest.mock() en la parte superior del archivo de prueba.

verbose [boolean]

Por defecto: false

Indica cuando cada prueba individual debería ser reportada durante la ejecución. Todos los errores, seguirán mostrándose al final, luego de la ejecución.

watchPathIgnorePatterns [array]

Por defecto: []

disponible en Jest 21.0.0+

Un areglo de patrones RegExp que son comparados con todas las rutas de archivo fuente antes de re-ejecutar pruebas en modo expectador. Si la ruta de archivo coincide con alguno de los patrones, cuando se actualice, ya no pondrá en marcha una re-ejecución de pruebas.

Estos patrones corresponden a la secuencia entera. Usa la cadena <rootDir> símbolo para incluir la ruta al directorio de raíz de su proyecto para evitar que ignore accidentalmente todos sus archivos en diferentes entornos que pueden tener directorios raíz diferentes. Example: ["<rootDir>/node_modules/"].
