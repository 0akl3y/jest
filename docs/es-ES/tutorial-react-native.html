Testing React Native Apps

En Facebook, utilizamos Jest para probar aplicaciones React Native.

Aprenda más sobre las pruebas a un ejemplo de app React Native funcional leyendo las siguiente series: Part 1: Jest – Snapshot come into play y Part 2: Jest – Redux Snapshots for your Actions and Reducers.

Setup

Empezando desde la versión 0.38 de react-native, una configuración de Jest siempre está incluída cuando se ejecuta react-native init. La siguiente configuración debería ser añadida automáticamente a su archivo package.json:

// package.json
  "scripts": {
    "test": "jest"
  },
  "jest": {
    "preset": "react-native"
  }

Nota: Si está mejorando su aplicación react-native y utilizó previamente la configuración preestablecida jest-react-native, quite la dependencia de su archivo package.json y cambie la configuración preestablecida a react-native en su lugar.

Simply run yarn test to run tests with Jest.

Test de Instantánea

Creemos una prueba de snapshot para un componente de introducción pequeño, con pocas visualizaciones, componentes de texto y algunos estilos:

// Intro.js
import React, {Component} from 'react';
import {StyleSheet, Text, View} from 'react-native';

const styles = StyleSheet.create({
  container: {
    alignItems: 'center',
    backgroundColor: '#F5FCFF',
    flex: 1,
    justifyContent: 'center',
  },
  instructions: {
    color: '#333333',
    marginBottom: 5,
    textAlign: 'center',
  },
  welcome: {
    fontSize: 20,
    margin: 10,
    textAlign: 'center',
  },
});

export default class Intro extends Component {
  render() {
    return (
      <View style={styles.container}>
        <Text style={styles.welcome}>Welcome to React Native!</Text>
        <Text style={styles.instructions}>
          This is a React Native snapshot test.
        </Text>
      </View>
    );
  }
}

Ahora utilizaremos el procesador de prueba de React y la función de instantánea de Jest para interactuar con el componente y capturar el resultado representado y crear un archivo de instantánea:

// __tests__/Intro-test.js
import React from 'react';
import Intro from '../Intro';

import renderer from 'react-test-renderer';

test('renders correctly', () => {
  const tree = renderer.create(<Intro />).toJSON();
  expect(tree).toMatchSnapshot();
});

When you run yarn test or jest, this will produce an output file like this:

// __tests__/__snapshots__/Intro-test.js.snap
exports[`Intro renders correctly 1`] = `
<View
  style={
    Object {
      "alignItems": "center",
      "backgroundColor": "#F5FCFF",
      "flex": 1,
      "justifyContent": "center",
    }
  }>
  <Text
    style={
      Object {
        "fontSize": 20,
        "margin": 10,
        "textAlign": "center",
      }
    }>
    Welcome to React Native!
  </Text>
  <Text
    style={
      Object {
        "color": "#333333",
        "marginBottom": 5,
        "textAlign": "center",
      }
    }>
    This is a React Native snapshot test.
  </Text>
</View>
`;

La próxima vez que ejecute pruebas, la salida renderizada será comparada a los snapshots previamente creados. El snapshot debería estar entregado junto a cambios en el código. Cuando falla una prueba de snapshot, necesita inspeccionar si es un cambio intencionado o no. Si el cambio es esperado, puede invocar Jest con jest -u para sobreescribir el snapshot existente.

El código para este ejemplo está disponible en examples/react-native.

Configuración preestablecida

La pre-instalación configura el entorno y es muy terco y se basa en lo que encontramos útil en Facebook. Todas las opciones de configuración pueden ser sobreescritas sólo así como pueden ser personalizadas cuando no se utiliza una configuración preestablecida.

Entorno

react-native ships with a Jest preset, so the jest.preset field of your package.json should point to react-native. La configuración preestablecida es un nódo de ambiente que imita al ambiente de una app React Native. Debido a que no carga ninguna DOM o APIs de buscador, mejora de gran manera el tiempo de inicio de Jest.

Personalización de transformIgnorePatterns

La opción transformIgnorePatterns puede ser utilizada para aprobar o rechazar archivos a ser transformado con babel. Desafortunadamente, muchos módulos npm react-native no pre-compilan su código fuente antes de publicar.

Por defecto, la configuración preestablecida de jest-react-native sólo procesa los archivos fuente del proyecto y react-native. Si usted tiene dependencias npm que deben ser transformadas, puede personalizar esta opción de configuración al colocar en lista de aprobación a los módulos que no sean react-native:

"transformIgnorePatterns": [
  "node_modules/(?!(react-native|my-project|react-native-button)/)"
]

setupFiles

Si desea proporcionar configuración adicional para cada archivo de prueba, la opción de configuraciónsetupFiles puede ser utilizada para especificar scripts de configuración.

moduleNameMapper

moduleNameMapper puede ser utilizada para mapear una ruta de módulo a un módulo diferente. Por defecto, la configuración preestablecida mapea todas las imagenes a un módulo de almacen de imágenes, pero si un módulo no puede ser encontrado esta opción de configuración puede ayudar:

"moduleNameMapper": {
  "my-module.js": "<rootDir>/path/to/my-module.js"
}

Consejos

Simulacros de módulos nativos usando jest.mock

La configuración preestablecida de Jest incluída en react-native viene con unas pocas simulaciones por defecto que son aplicadas en un repositorio react-native. Sin embargo, algunos componentes o componentes de terceros de react-native dependen de código nativo para ser renderizados. En estos casos, el sistema de simulación manual de Jest puede ayudar a simular la implementación subyacente.

Por ejemplo, si su código depende de un componente de video nativo de un tercerdo llamado react-native-video puede que quiera almacenarlo con una simulación manual de esta manera:

jest.mock('react-native-video', () => 'Video');

Esto renderizará el componente como <Video {...props} /> con todos sus apoyos en la salida del snapshot. See also caveats around Enzyme and React 16.

A veces es necesario proporcionar un "mock" manual más complejo. Por ejemplo, si le gustaría hacerle llegar a una simulación los tipos de apoyo o campos estáticos de un componente nativo, usted puede devolver un componente React diferente a una simulación a través de este ayudante de jest-react-native:

jest.mock('path/to/MyNativeComponent', () => {
  const mockComponent = require('react-native/jest/mockComponent');
  return mockComponent('path/to/MyNativeComponent');
});

O si prefieres crear tu propio "mock" manual, puedes hacer algo así:

jest.mock('Text', () => {
  const RealComponent = require.requireActual('Text');
  const React = require('React');
  class Text extends React.Component {
    render() {
      return React.createElement('Text', this.props, this.props.children);
    }
  }
  Text.propTypes = RealComponent.propTypes;
  return Text;
});

En otros casos quizás quiera simular un módulo nativo que no sea un componente React. Se puede aplicar la misma técnica. Recomendamos inspeccionar el código fuente del módulo nativo y logear el módulo cuando ejecute react native app en un dispositivo real, luego modelar una simulación manual después del módulo real.

Si termina simulando los mismo módulos una y otra vez, se recomienda que defina estas simulaciones en un archivo separado y las añada a la lista de setupFiles.

@providesModule

Si le gustaría utilizar el sistema de módulo @providesModule de Facebook a través de un paquete npm, la opción de configuración, por defecto, de velocidad deberá ser sobreescrita y los módulos npm deben ser añadidos a providesModuleNodeModules:

"haste": {
  "defaultPlatform": "ios",
  "platforms": ["android", "ios"],
  "providesModuleNodeModules": [
    "react",
    "react-native",
    "my-awesome-module",
    "my-text-component"
  ]
},

Si le gustaría probar un plataforma, por defecto, diferente o si está construyendo para otras plataformas, la opción de configuración defaultPlatform y platforms puede ser actualizada.
