Setup and Teardown

Muchas veces mientras escribes pruebas necesitas hacer algún trabajo de configuración antes de ejecutar las pruebas, y uno de finalización luego de éstas. Jest provee funciones de ayuda para manejar esto.

Repitiendo "preparación" para muchas pruebas

Si hay algún trabajo que necesitas hacer repetidamente para varias pruebas, puedes utilizar beforeEach y afterEach.

Por ejemplo, digamos que varias pruebas interactúan con la base de datos de ciudades. Existe un método initializeCityDatabase() que debe ser llamado antes de cada una de estas pruebas, y otro clearCityDatabase() que debe ser llamado luego de cada prueba. Puedes hacer esto con:

beforeEach(() => {
  initializeCityDatabase();
});

afterEach(() => {
  clearCityDatabase();
});

test('city database has Vienna', () => {
  expect(isCity('Vienna')).toBeTruthy();
});

test('city database has San Juan', () => {
  expect(isCity('San Juan')).toBeTruthy();
});

beforeEach y afterEach pueden manejar un código asíncrono de la misma forma que las pruebas manejan el código asíncrono - pueden tomar el parámetro done o devolver una promesa. Por ejemplo, si initializeCityDatabase() devuelve una promesa que es resuelta cuando la base de datos fue iniciada, nos gustaría que retornara una promesa:

beforeEach(() => {
  return initializeCityDatabase();
});

Preparación de una vez

En algunos casos, solo necesitarás realizar la preparación una sola vez, al principio del archivo. Esto puede ser particularmente molesto cuando la configuración es asíncrona, así que simplemente puedes hacerlo en línea. Jest proporciona beforeAll y afterAll para manejar esta situación.

Por ejemplo, si ambos initializeCityDatabase y clearCityDatabase devolvió promesas, y la base de datos de la ciudad puede ser rehusada entre pruebas, podríamos cambiar el código de nuestra prueba a:

beforeAll(() => {
  return initializeCityDatabase();
});

afterAll(() => {
  return clearCityDatabase();
});

test('city database has Vienna', () => {
  expect(isCity('Vienna')).toBeTruthy();
});

test('city database has San Juan', () => {
  expect(isCity('San Juan')).toBeTruthy();
});

Alcance

Por defecto, los bloques before y after aplicarán a cada prueba en un archivo. Además también puedes agrupar pruebas usando un bloque describe. Cuando se encuentran dentro de un bloque describe, los bloques before y after sólo aplican a las pruebas de ese bloque describe.

Por ejemplo, digamos que no teníamos sólo una ciudad de base de datos, sino también una de comida. Podríamos hacer diferentes configuraciones para distintas pruebas:

// Applies to all tests in this file
beforeEach(() => {
  return initializeCityDatabase();
});

test('city database has Vienna', () => {
  expect(isCity('Vienna')).toBeTruthy();
});

test('city database has San Juan', () => {
  expect(isCity('San Juan')).toBeTruthy();
});

describe('matching cities to foods', () => {
  // Applies only to tests in this describe block
  beforeEach(() => {
    return initializeFoodDatabase();
  });

  test('Vienna <3 sausage', () => {
    expect(isValidCityFoodPair('Vienna', 'Wiener Schnitzel')).toBe(true);
  });

  test('San Juan <3 plantains', () => {
    expect(isValidCityFoodPair('San Juan', 'Mofongo')).toBe(true);
  });
});

Note que el nivel más alto beforeEach es ejecutado antes de beforeEach dentro del bloque describe. Puede ayudar a ilustrar la orden de ejecución de todos los ganchos.

beforeAll(() => console.log('1 - beforeAll'));
afterAll(() => console.log('1 - afterAll'));
beforeEach(() => console.log('1 - beforeEach'));
afterEach(() => console.log('1 - afterEach'));
test('', () => console.log('1 - test'));
describe('Scoped / Nested block', () => {
  beforeAll(() => console.log('2 - beforeAll'));
  afterAll(() => console.log('2 - afterAll'));
  beforeEach(() => console.log('2 - beforeEach'));
  afterEach(() => console.log('2 - afterEach'));
  test('', () => console.log('2 - test'));
});

// 1 - beforeAll
// 1 - beforeEach
// 1 - test
// 1 - afterEach
// 2 - beforeAll
// 1 - beforeEach
// 2 - beforeEach
// 2 - test
// 2 - afterEach
// 1 - afterEach
// 2 - afterAll
// 1 - afterAll

Orden de ejecución de descripción y bloques de pruebas

Jest ejecuta todos los controles de descripción en un archivo de prueba before que ejecuta cualquiera de las pruebas actuales. Esta es otra razón para configurar y derribar en before* y after* controladores en vez de en los bloques de descripción. Una vez que los bloques de descripción están completos, por defecto Jest corre todas las pruebas en serie en el orden que fueron halladas en la fase de colección, esperando que cada una termine y sea arreglada antes de avanzar.

Considere el siguiente archivo de prueba ilustrativa y salida:

describe('outer', () => {
  console.log('describe outer-a');

  describe('describe inner 1', () => {
    console.log('describe inner 1');
    test('test 1', () => {
      console.log('test for describe inner 1');
      expect(true).toEqual(true);
    });
  });

  console.log('describe outer-b');

  test('test 1', () => {
    console.log('test for describe outer');
    expect(true).toEqual(true);
  });

  describe('describe inner 2', () => {
    console.log('describe inner 2');
    test('test for describe inner 2', () => {
      console.log('test for describe inner 2');
      expect(false).toEqual(false);
    });
  });

  console.log('describe outer-c');
});

// describe outer-a
// describe inner 1
// describe outer-b
// describe inner 2
// describe outer-c
// test for describe inner 1
// test for describe outer
// test for describe inner 2

Aviso General

Si una prueba está fallando, una de las primeras cosas por chequear debería ser si está fallando cuando es la única prueba que se corre. En Jest es simple ejecutar sólo una prueba - solo cambia temporalmente el comando test a test.only:

test.only('this will be the only test that runs', () => {
  expect(true).toBe(false);
});

test('this test will not run', () => {
  expect('A').toBe('A');
});

Si tienes una prueba que falla seguidamente cuando se corre como parte de una serie más grande, pero no falla cuando cuando se corre sólo, es probable que algo de una prueba distinta esté interfiriendo con ésta. Puedes arreglar esto limpiando algo de estado compartido con beforeEach. Si no estás seguro de que algún estado compartido está siendo modificado, puedes tratar también beforeEach que simplemente genera datos.
