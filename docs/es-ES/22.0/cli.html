Jest CLI Options

El comando jest tiene varias opciones útiles. Puede ejecutar jest --help para ver todas las opciones disponibles. Muchas de las opciones mostradas abajo tambien pueden ser usadas juntas para ejecutar pruebas exactamente como usted desee. Cada una de las opciones de Configuración de Jest pueden ser especificadas a través de la línea de comandos CLI.

A continuación se muestra un breve resumen:

Ejecución desde la línea de comandos

Ejecuta todos las pruebas (por defecto):

jest

Ejecuta sólo las pruebas que se especificaron con un patrón o un nombre de archivo:

jest my-test #or
jest path/to/my-test.js

Ejecuta las pruebas relacionadas con los archivos modificados basados en hg/git (archivos no confirmados):

jest -o

Ejecuta las pruebas relacionadas con path/to/fileA.js y path/to/fileB.js:

jest --findRelatedTests path/to/fileA.js path/to/fileB.js

Run tests that match this spec name (match against the name in describe or test, basically).

jest -t name-of-spec

Ejecuta el modo "en la mira":

jest --watch #runs jest -o by default
jest --watchAll #runs all tests

El modo espectador también permite especificar el nombre o ruta hacia un archivo para centrarse en un conjunto específico de pruebas.

Uso con scripts npm

Si ejecuta Jest a través de npm test, aun puede usar los argumentos de la linea de comandos insertando un -- between npm test y los argumentos de Jest. En lugar de:

jest -u -t="ColorPicker"

puedes usar:

npm test -- -u -t="ColorPicker"

jest <regexForTestFiles>

Cuando ejecute jest con un argumento, ese argumento es tratado como una expresión regular para coincidir con archivos en su proyecto. Es posible ejecutar test suites proporcionando un patrón. Sólo los archivos que el patrón logre coincidir serán seleccionados y ejecutados. Note: depending on your terminal, you may need to quote this argument: jest "my.*(complex)?pattern".

--bail

Alias: -b. Finaliza la suit de pruebas inmediatamente luego de la primera prueba fallada.

--cache

Si deseas usar la caché. Por defecto es verdadero. Desactive el caché usando --no-cache. Nota: el caché debería ser solamente desactivado si usted experimenta problemas relacionados al mismo. En promedio, desactivar el caché hace a Jest al menos dos veces más lento.

Si desea insperccionar el caché, utilice --showConfig y observe el valor de cacheDirectory. Si necesita borrar el caché, utilice --clearCache.

--changedFilesWithAncestor

When used together with --onlyChanged or --watch, it runs tests related to the current changes and the changes made in the last commit.

--ci

Cuando se activa esta opción, Jest asumirá que está siendo ejecutado en un entorno de Integración Continua (CI). Esto cambia el comportamiento cuando se encuentra un nuevo snapshot. En lugar del comportamiento regular de guardar un nuevo snapshot automáticamente, fallará la prueba y requerirá a Jest que ejecute con --updateSnapshot.

--limpiarCaché

disponible en Jest 22.0.0+

Borra el directorio caché de Jest y se sale sin ejecutar pruebas. Borrará cacheDirectory si la opción pasa, o el directorio caché por defecto de Jest. El directorio caché por defecto puede ser encontrado llamando a jest --showConfig. Nota: vaciar el caché reducirá el rendimiento.

--collectCoverageFrom=<glob>

En relación al directorio de raíz, el patrón global coincide con los archivos de los que debe recopilarse la información de cobertura.

--colors

Resultados forzados de pruebas se muestran resaltados incluso si stdout no es un TTY.

--config=<path>

Alias: -c. La ruta hacia un archivo de configuración de Jest especificando cómo encontrar y ejecutar pruebas. Si ningún rootDir está colocado en la configuración, se asume que el directorio actual es el rootDir para el proyecto. Esto también puede ser un valor JSON-encoded el cual Jest usará como configuración.

--coverage

Indica que la información de la cobertura de prueba debería ser recolectada y reportada en la salida.

--debug

Imprime información de debugging sobre su configuración de Jest.

--env=<environment>

El entorno de prueba utilizado para todas las pruebas. Esto puede apuntar a cualquier archivo o módulo de nodo. Ejemplos: jsdom, node or path/to/my-environment.js.

--expand

Alias: -e. Usa esta opción para mostrar todas las diferencias y errores en vez de un parche.

--findRelatedTests <spaceSeparatedListOfSourceFiles>

Encuentra y ejecuta las pruebas que cubran una lista de espacio separado de archivos fuente que fueron pasados como argumentos. Útil para integraciones hook "pre-commit" para ejecutar la cantidad mínima de pruebas necesarias.

--forceExit

Fuerza a Jest a salir después de que todos las pruebas completen su ejecución. Esto es útil cuando los recursos dispuestos por el código de prueba no puedan ser limpiados adecuadamente. Nota: Esta característica es un "escape-hatch". If Jest doesn't exit at the end of a test run, it means external resources are still being held on to or timers are still pending in your code. Se aconseja remover los recursos externos luego de cada prueba para asegurarse de que Jest pueda cerrar limpiamente.

--help

Muestra la información de ayuda, similar a esta página.

--json

Imprime los resultados de la prueba en JSON. Este modo enviará todos los otros resultados de la prueba y los mensajes de usuario a stederr.

--outputFile=<filename>

Escribe los resultados de las pruebas en un archivo cuando la opción --json es también especificada.

--lastCommit

When used together with --onlyChanged, it will run all tests affected by file changes in the last commit made.

--listTests

Pone en lista todas la pruebas como JSON que Jest ejecutará en base a los argumentos y salidas. Esto pueden utilizarse junto a --findRelatedTests para saber qué pruebas Jest ejecutará.

--logHeapUsage

Registra el uso de la pila después de cada prueba. Útil para debuggear leaks de memoria. Utilice junto a --runInBand y --expose-gc en nodo.

--maxWorkers=<num>

Alias: -w. Especifica el número máximo de trabajadores que la piscina de trabajadores creará para ejecutar pruebas. Esto predetermina la cantidad de nuclueos disponibles en su maquina. Puede ser útil ajustar esto en entornos de recursos limitados como IC, pero el valor predeterminado debería ser adecuado para la mayoría de los casos de uso.

--noStackTrace

Desactiva el stack trace de la salida de los resultados de las pruebas.

--notify

Activa notificaciones para los resultados de las pruebas. Bueno para cuando no desee que su conciencia se pueda concentrar en otra cosa excepto en pruebas de JavaScript.

--onlyChanged

Alias: -o. Intenta identificar qué pruebas ejecutar basado en cuáles archivos han cambiado en el repositorio actual. Sólo funciona si usted está ejecutando pruebas en un repositorio de git/hg en ese momento y requiere un grafo de dependencia estática (es decir, no requiere dinámica).

--passWithNoTests

Permite al suite de pruebas pasar cuando no se consigan archivos.

--projects <project1> ... <projectN>

Ejecuta tests de uno o más proyectos.

--runInBand

Alias: -i. Ejecuta todas las pruebas en serie en el proceso presente, en lugar de crear una piscina de trabajadores de child process que ejecuten pruebas. Esto puede ser útil para debuggear.

--setupTestFrameworkScriptFile=<archivo>

La ruta a un modulo que ejecuta algún código para configurar o preparar el marco de referencia de prueba antes de cada prueba. Tenga en cuenta los archivos importados por el script preparado no será burlado durante la prueba.

--showConfig

Imprime tu configuración Jest luego finaliza el programa.

--silent

Previene a las pruebas de imprimir mensajes a través de la consola.

--testNamePattern=<regex>

Alias: -t. Ejecuta únicamente pruebas y suites de prueba con un nombre que coincida con el regex. Por ejemplo, suponga que quiere ejecutar únicamente pruebas relacionadas con autorización las cuales tendrán nombres como "GET /api/posts with auth", entonces podrá utilizar jest -t=auth.

--testLocationInResults

Añade un campo de location a los resultados de la prueba. Útil si desea reportar la localización de una prueba en un reportador.

Note that column is 0-indexed while line is not.

{
  "column": 4,
  "line": 5
}

--testPathPattern=<regex>

Una cadena de patron regexp que se compara con todas las rutas de prueba antes de ejecutar la prueba.

--testRunner=<ruta>

Te permite especificar un corredor de prueba personalizado.

--updateSnapshot

Alias: -u. Use esta bandera para re-grabar cada snapshot que falle durante esta prueba de funcionamiento. Puede usarse junto a un patrón de suite de prueba o con --testNamePattern para re-grabar snapshots.

--useStderr

Desvía toda la salida a stderr.

--verbose

Muestra los resultados de la prueba individual con la jerarquía de la suite de prueba.

--version

Alias: -v. Imprime la versión y finaliza.

--watch

Vigile archivos para pruebas de cambio o re-run relacionadas con archivos cambiados. Si desea hacerle re-run a todas las pruebas cuando un archivo ha cambiado, use la opción --watchAll.

--watchAll

Vigile archivos de los cambios y haga re-run de todas las pruebas cuando algo cambie. Si desea hacer solamente re-run de pruebas que dependan de los archivos cambiados, utilice la opción --watch.

--watchman

Whether to use watchman for file crawling. Defaults to true. Disable using --no-watchman.
